# Codex Portable Baseline Design

## Goal

Manage a portable, user-authored Codex baseline with GNU Stow while leaving authentication, sessions, caches, downloaded dependencies, and machine-specific runtime state in `~/.codex`.

## Package layout

The repository will gain one Stow package. Its paths mirror the user target:

```text
codex/
└── .codex/
    ├── AGENTS.md
    ├── config.toml
    ├── prompts/
    │   └── <authored prompt files>
    └── skills/
        └── <authored skill directories>
```

With the repository's `--no-folding` configuration, Stow creates `~/.codex` as a normal directory and individually symlinks the managed files and directories. Codex can continue to write its local runtime state alongside those links.

## Managed content

### `AGENTS.md`

Track the existing global instructions at `~/.codex/AGENTS.md` unchanged.

### `config.toml`

Create a curated baseline from the live configuration using an allowlist. The baseline contains only:

- top-level durable preferences: `personality`, `model`, and `model_reasoning_effort`;
- explicit feature preferences under `[features]` and `[memories]`;
- each explicitly enabled plugin entry under `[plugins]`;
- a future MCP-server block only when its command, arguments, working directory, and environment are portable and contain no credentials.

The baseline excludes:

- all `[projects.*]` trust records;
- `[tui.*]`, `[desktop]`, `[hooks.state]`, and `[marketplaces.*]` state;
- the current `node_repl` and `computer-use` MCP-server blocks, because they contain installation-specific paths and runtime values;
- `notify`, because the current value invokes a bundled macOS executable through an installation-specific absolute path;
- `[shell_environment_policy]`, including its environment settings;
- secrets, access tokens, API keys, passwords, and credentials of every kind.

The Stow-managed file is the source of truth. Machine-specific Codex state is intentionally regenerated by Codex rather than merged into it.

### Prompts and skills

Track only files deliberately authored by the user:

- `~/.codex/prompts/*.md` for reusable custom prompts;
- `~/.codex/skills/<name>/` for reusable custom skills, including their required `SKILL.md` and supporting files.

Do not add placeholder files for the currently empty `prompts/` directory or the current system-provided `skills/` content. Add paths when authored content exists.

## Explicitly local content

Never Stow or commit the following:

- credentials and identity: `auth.json`, `installation_id`;
- conversations and user data: `sessions/`, `archived_sessions/`, `history.jsonl`, `attachments/`, and `shell_snapshots/`;
- caches and downloaded application content: `cache/`, `plugins/cache/`, `plugins/.plugin-appserver/`, `vendor_imports/`, `packages/`, and `computer-use/`;
- generated state: SQLite databases and their `-wal`/`-shm` files, `state_*.sqlite`, `logs_*.sqlite`, `goals_*.sqlite`, `memories_*.sqlite`, `models_cache.json`, and `session_index.jsonl`;
- machine-local metadata: `version.json`, `.personality_migration`, `.codex-global-state.json`, `.tmp/`, `ipc/`, and `process_manager/`.

## Migration and verification

1. Back up the existing live `AGENTS.md` and `config.toml` before changing their locations.
2. Create the `codex/.codex/` package and write the reviewed curated configuration there.
3. Move the existing `AGENTS.md` into that package without changing its contents.
4. Preview Stow with `stow -n codex -t ~`; it must report only the two planned links.
5. Create the real symlinks with `stow codex -t ~` and verify both targets resolve into the dotfiles repository.
6. Start Codex and confirm that it reads the baseline and can still create local runtime state.
7. Commit only the `codex/` package and its documentation.

## Success criteria

- `~/.codex/config.toml` and `~/.codex/AGENTS.md` are symlinks into the `codex` Stow package.
- The repository contains no credentials, histories, databases, caches, or downloaded Codex binaries.
- A new machine can restore the same global instructions and baseline preferences with `stow codex -t ~`.
- Existing and future Codex runtime state remains writable under the real `~/.codex` directory.
